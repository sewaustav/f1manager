# Драфт (Фаза 1) — дизайн

Дата: 2026-07-09

## Цель

Реализовать веб-версию драфта F1 Manager: пошаговый выбор команды/пилота/тим-принципала
несколькими игроками группы через WebSocket-уведомления + HTTP-пики, с автозаполнением
команд-ботов после завершения. Строится против интерфейса `DynamicRepo` (расширенного),
с in-memory реализацией для тестов. Реальный Redis/PG — отдельная фаза.

## Решения, принятые с пользователем

- **Порядок работ**: драфт-первым, против интерфейса DynamicRepo; реальное хранилище — Фаза 3.
- **Порядок ходов**: ротация влево (как в примере): `[1,2,3,4]→[2,3,4,1]→[3,4,1,2]→[4,1,2,3]`. Не классическая змейка.
- **Таймаут хода**: нет — ждём игрока бесконечно.
- **Экономика драфта**: как в CLI. Виртуальный бюджет игрока 110 млн до выбора команды;
  при выборе команды бюджет пересчитывается: `team.Budget − уже_потрачено`.
- **Старт**: организатор через HTTP (`POST /draft/start`); первичный порядок — случайная перестановка.

## Правила драфта

- **4 раунда**, в каждом игрок делает **1 пик**: команда, пилот или тим-принципал (порядок внутри раунда произвольный).
- **Лимиты ростера**: ≤ 1 команда, ≤ 2 пилота, ≤ 1 тим-принципал.
  Т.к. 4 раунда × 1 пик = ровно 4 слота `{1 команда, 2 пилота, 1 принципал}`,
  лимиты автоматически принуждают к валидному ростеру.
- **Игроков**: от 1 до 11.
- **Выбор команды** сопровождается обязательным выбором мотора:
  - Заводская команда (`Manufacture`) с чужим мотором → мотор принудительно переназначается на собственный.
  - Стоимость мотора: завод — `price`; полу-завод-клиент и клиент (`Semi`/`Client`) — `price + 10`
    (логика `internal/cli/initial.go → chooseEngine()`).

## Конечный автомат (DraftDispatcher)

Состояние на группу, защищено mutex; живёт в памяти (как setup-диспетчер). При рестарте драфт теряется — приемлемо для Фазы 1.

- **Порядок ходов**: `order_r = leftRotate(baseOrder, r)` для `r = 0..3`. Полная очередь = 4 раунда подряд, всего `4 × N` ходов. `baseOrder` — случайная перестановка игроков группы.
- **`StartDraft(ctx, groupID, players)`**: тасует игроков, строит очередь, инициализирует бюджет каждого игрока = 110 млн, шлёт `draft_turn` первому, помечает драфт активным.
- **`SubmitPick(ctx, userID, groupID, pick)`**: проверяет, что сейчас ход игрока → применяет пик через сервис → двигает указатель → `draft_turn` следующему; если очередь кончилась — автозаполнение + broadcast `draft_finished`.
- **Таймаута нет**. Разрыв соединения: указатель стоит; игрок переподключается и ходит. Не его ход → `409`.
- **Конкурентность**: как в setup-диспетчере — состояние группы под mutex, гарантия единственного продвижения указателя.

## Пики, лимиты и экономика

DTO: `Draft{ Pick DraftItem; ItemID int64; Engine *models.ICEName }` (для команды `Engine` обязателен).
`DraftItem`: `DraftPilot`, `DraftTeam`, `DraftPrincipal` (уже есть в dto).

Валидация каждого пика:
1. Ход игрока (диспетчер).
2. Лимиты ростера (см. правила).
3. Доступность элемента: команда без владельца; пилот свободен; принципал свободен.
4. Бюджет:
   - Пилот: `price − sponsors`; принципал: `price`.
   - Мотор при выборе команды: клиент/полу — `price + 10`, завод — `price`, мотор завода принудительно свой.
   - **Источник правды по деньгам — репозиторий.** До команды бюджет = 110.
     При выборе команды: `newBudget = team.Budget − (110 − currentBudget)`, далее вычитается стоимость мотора.

Каждый пик **сразу применяется в DynamicRepo** (назначение команды/пилота/принципала/мотора + бюджет),
как каждый трансфер в CLI. Диспетчер в памяти держит только порядок ходов и счётчики ростера.

## Автозаполнение после драфта (автоматически при завершении)

- Свободные пилоты (`garage = null`) распределяются командам, которым не хватает до 2 пилотов, **по убыванию рейтинга**.
- Команды-боты (без игрока) получают дефолтных пилотов и настройки.
- Ручка `POST /draft/bots/swap` — организатор меняет пилотов **только между командами-ботами** (команды игроков не трогаются).

## Транспорт

- `POST /api/v1/draft/start` — организатор (в Фазе 1: любой член группы, как текущий InitRound — ограничение, ролей пока нет).
- `POST /api/v1/draft/pick` — тело `Draft` DTO.
- `POST /api/v1/draft/bots/swap` — тело `{team_a, team_b, pilot_a, pilot_b}`.
- WS клиенту: `draft_turn {round, your_turn}`, `draft_pick_made {user_id, pick, item_id}` (broadcast), `draft_finished`.
- Ошибки: не твой ход → 409; невалидный пик (занято/лимит/бюджет) → 400 с причиной; элемент не найден → 404; драфт не начат/завершён → 409.

## Расширение DynamicRepo

Добавить минимум, переиспользуя существующее где можно. Ориентировочный набор (точные сигнатуры — на этапе плана, сверив с текущими `ExecutePilotTransfer` и т.п.):

- `AssignTeamToPlayer(ctx, userID, groupID, teamID)` — назначить команду игроку.
- `SetPlayerBudget(ctx, userID, groupID, budget)` — установить абсолютный бюджет (init 110 + пересчёт).
- `AssignPilotToTeam(ctx, pilotID, teamID, groupID)` — назначить пилота в гараж команды (или reuse `ExecutePilotTransfer`).
- `SetPlayerPrincipal(ctx, userID, groupID, principalID)` — назначить принципала (или reuse `ExecutePrincipalTransfer`).
- `SetTeamEngine(ctx, teamID, groupID, ice)` — установить мотор (или reuse `UpdateTeam`).
- `GetUnassignedPilots(ctx, groupID)` — свободные пилоты (garage null).
- `GetBotTeams(ctx, groupID)` — команды без игрока.

Заглушка `stub` расширяется теми же методами (`not implemented`), in-memory реализация — рабочая (для тестов).

## Тестирование

- **Unit**:
  - генерация порядка (ротация влево, N игроков, 4 раунда, полная очередь `4×N`);
  - валидация пиков: лимиты ростера, доступность, бюджет, правила мотора + принудительное переопределение для завода, пересчёт бюджета при выборе команды;
  - автозаполнение: свободные пилоты по убыванию рейтинга, дефолты ботов, `bots/swap` только между ботами.
- **Интеграция**: полный цикл драфта на in-memory DynamicRepo + фейковый Notifier/сессии,
  несколько игроков, проверка валидности итоговых ростеров и корректности экономики.
